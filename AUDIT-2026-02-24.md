# WARREN / IGW CODE AUDIT REPORT

**Date**: 2026-02-24
**Branch**: `v3.0`
**Scope**: `Framework/v3/warren/`, `Games/igw/`, `Framework/v3/igw/`, `Framework/v3/warren-sdk/`, `Framework/v3/shared/`

---

## EXECUTIVE SUMMARY

The Warren Framework and IGW codebase contain **~130 active files** powering a working procedural dungeon/terrain generation system with chunked world loading, multi-biome terrain, and a server-CSG minimap. The architecture successfully applies FBP, Houdini SOP, and React vDOM paradigms to Roblox game development.

However, the codebase carries **~98,500 lines of dead code** across abandoned framework versions, removed subsystems still loaded at boot, and a legacy component tree that no project.json references. Two critical architectural violations — a bootstrap divergence that gates `_G.Warren` differently across environments, and a `shared.dungeonReady` schema mismatch between write sites — represent latent production bugs. The dual-runtime principle (Roblox + Lune convergence) is violated by 4 files that execute Roblox API calls at module load time, blocking Lune require paths.

**By the numbers:**

| Metric | Count |
|--------|-------|
| Active files | ~130 |
| Dead files | ~183 |
| Dead lines of code | ~98,488 |
| Critical violations | 2 |
| High-severity violations | 5 |

## SECTION 2: ARCHITECTURAL VIOLATIONS

### CRITICAL

#### V-2.1: `_G.Warren` Bootstrap Divergence
**Invariant violated**: Closure encapsulation, three-domain boundary
**Files**: `Games/igw/Bootstrap.server.lua`, `Framework/v3/igw/Bootstrap.server.lua`

The two bootstraps set `_G.Warren` under different conditions:

- `Games/igw/Bootstrap.server.lua`: Sets `_G.Warren` **only in Studio** (`if RunService:IsStudio()`). In production, `_G.Warren` is `nil`.
- `Framework/v3/igw/Bootstrap.server.lua`: Sets `_G.Warren` **unconditionally**.

Meanwhile, **20+ pipeline files** (~55 access sites) read `_G.Warren.Dom`, `_G.Warren.ClassResolver`, `_G.Warren.Styles`, `_G.Warren.System.Debug` — all of which will be `nil` in production if the `Games/igw/` bootstrap is active.

This works today only because the pipeline runs on the Lune VPS (where `lune/mapgen/init.lua` sets its own mock `_G.Warren`). If any pipeline node ever runs on a production Roblox server, it will crash.

**Affected files** (sample): `DungeonOrchestrator.lua`, `MaterializerNode.lua`, `ShellBuilder.lua`, `LightBuilder.lua`, `TrussBuilder.lua`, `DoorCutter.lua`, `ShoreCutterNode.lua`, `RockScatterNode.lua`, `WorldClient.lua`, `WorldMapOrchestrator.lua`, and ~10 more.

#### V-2.2: `shared.dungeonReady` Schema Mismatch
**Invariant violated**: Signal discipline
**File**: `Games/igw/src/WorldMapOrchestrator.lua`

Two write sites with incompatible schemas:

- **Line ~509** (non-world-map path): `shared.dungeonReady = { spawn = spawnData }`
- **Line ~672** (world-map path): `shared.dungeonReady = true`

Any consumer checking `shared.dungeonReady.spawn` will crash on the world-map path. Any consumer checking `shared.dungeonReady == true` will get a table on the non-world-map path.

Additionally, the consumer (`Bootstrap.server.lua`) uses a **busy-loop poll** (`while not shared.dungeonReady do task.wait(0.1) end`) instead of a proper signal.

### HIGH

#### V-2.3: Module-Load-Time Roblox API Calls
**Invariant violated**: Dual-runtime awareness
**Files**:
- `Games/igw/src/Pipeline/PortalBlender.lua` (lines 15-27)
- `Games/igw/src/Pipeline/PortalRoomBuilder.lua` (lines 21-33)
- `Framework/v3/warren/src/Node.lua` (top level)
- `Framework/v3/warren/src/Internal/SchemaValidator.lua` (line 111)

These files execute Roblox API calls at `require()` time, before any handler body runs:

- **PortalBlender + PortalRoomBuilder**: Both probe `workspace.Terrain:GetMaterialColor()` at module load to build a material validation table. This is also **duplicated code** (identical 12-line block).
- **Node.lua**: `game:GetService("CollectionService")` at module scope.
- **SchemaValidator.lua**: `local Players = game:GetService("Players")` — never used (dead import), but blocks Lune require.

#### V-2.4: Transport/State Still Loaded at Boot
**Invariant violated**: Dead code removal (ADR-005)
**File**: `Framework/v3/warren/src/init.lua`

Despite ADR-005 explicitly removing Transport and State, `init.lua` still `require()`s both subsystems unconditionally. They ship in the build, consume memory, and register unused IPC handlers.

#### V-2.5: Game Code Reads Framework Private State
**Invariant violated**: Three-domain boundary, bottom-up dependency
**Files**:
- `Games/igw/Bootstrap.server.lua` — writes `Warren.OpenCloud._robloxConfig`
- `Framework/v3/igw/src/Components/RegionManager.lua` (legacy) — reads `Warren.OpenCloud._robloxConfig`

Game code directly manipulates an undocumented `_robloxConfig` field on the OpenCloud module. This couples game code to framework internal state.

#### V-2.6: `Out.Fire` Monkey-Patching
**Invariant violated**: Signal discipline, node autonomy
**Files**: 7 files, ~12 call sites

Orchestrators and pipeline nodes overwrite `Out.Fire` on child nodes to intercept completion signals. Examples:
- `DungeonOrchestrator.lua` — patches `Out.Fire` on spawned pipeline nodes during `_syncCall`
- Legacy `DungeonOrchestrator.lua` — patches 5 child nodes at setup time

While this is an intentional pattern for the `_syncCall` mechanism, it violates FBP signal discipline (signals should flow through declared output ports, not be intercepted via monkey-patching).

#### V-2.7: Direct Cross-Node `In.onXyz()` Calls
**Invariant violated**: Node autonomy (FBP black-box principle)
**Files**: ~25 call sites across orchestrators and components

Orchestrators call `nodeInstance.In.onSomeHandler(nodeInstance, data)` directly instead of wiring through IPC. This creates tight coupling between orchestrator and node internal interface. Examples:
- `DungeonOrchestrator._syncCall()` — calls `node.In[handler](node, payload)`
- `WorldMapOrchestrator` — calls handler methods directly on child nodes

**Note**: This is the documented hub-and-spoke pattern. The violation is architectural (bypasses IPC) but intentional.

### MEDIUM

#### V-2.8: `WorldClient` Cross-Requires Pipeline Node
**File**: `Games/igw/src/WorldClient.lua`

Directly `require()`s `TerrainPainterNode` to call `computeHeightField()`. This creates a lateral dependency between nodes (violates bottom-up dependency and node autonomy). The height field computation should be a shared utility, not accessed by reaching into another node.

#### V-2.9: Node.lua `_instance` Private Field Access
**File**: `Framework/v3/igw/src/Components/DomBuilder.lua` (legacy, dead)

Accesses `node._instance` — a private DOM implementation field. Low priority since the file is dead code.

#### V-2.10: Unguarded SDK Roblox Globals
**Files**: `warren-sdk/Auth.lua`, `warren-sdk/Http.lua`

Use `task.spawn`, `task.wait`, `task.cancel`, `HttpService` without runtime guards. The SDK is Roblox-only by design, but if it ever needs to work in Lune tests, these will fail.

## SECTION 3: NAMING CONVENTION VIOLATIONS

### 3.1 Node Suffix Inconsistency

The pipeline uses two naming patterns interchangeably:

**With `Node` suffix** (10 files):
`InventoryNode`, `SplinePlannerNode`, `BlockoutNode`, `TerrainPainterNode`, `ShoreCutterNode`, `RockScatterNode`, `CaveCutterNode`, `MeshTerrainPainterNode`, `FloodFillNode`, `MiniMapNode`

**Without `Node` suffix** (20 files):
`ShellBuilder`, `LightBuilder`, `TrussBuilder`, `DoorCutter`, `DoorPlanner`, `Materializer`, `PortalBlender`, `PortalRoomBuilder`, `MountainBuilder`, `MountainRoomMasser`, `MountainTerrainPainter`, `MountainLightBuilder`, `WorldClient`, `DungeonOrchestrator`, `WorldMapOrchestrator`, etc.

No consistent rule determines which files get the suffix.

### 3.2 Name Collisions Across Directories

5 file names exist in **both** `Games/igw/src/Pipeline/` and `Framework/v3/igw/src/Components/`:

| Name | Active (Games/) | Dead (Framework/) |
|------|-----------------|-------------------|
| `DoorCutter` | Pipeline node | Component |
| `ShellBuilder` | Pipeline node | Component |
| `LightBuilder` | Pipeline node | Component |
| `TrussBuilder` | Pipeline node | Component |
| `DungeonOrchestrator` | Orchestrator | Component |

While the Framework copies are dead, the collisions create confusion during code navigation and grep searches.

### 3.3 `WorldClient` Misnomer
**File**: `Games/igw/src/WorldClient.lua`

This is a **server-side** node that manages world lifecycle (VPS communication, chunk loading, height field computation). The "Client" suffix implies client-side code. Should be `WorldServer` or `WorldBridge`.

### 3.4 Signal Name Fragmentation

Three distinct completion signal conventions across the codebase:

| Signal | Used By |
|--------|---------|
| `nodeComplete` | Most pipeline nodes (e.g., ShellBuilder, LightBuilder, TrussBuilder) |
| `buildPass` / `buildComplete` | DoorCutter, Materializer |
| `chunkDone` | DungeonOrchestrator chunk pipeline |
| `buildWorld` / `worldReady` | WorldMapOrchestrator <-> WorldClient |
| `dungeonReady` | WorldMapOrchestrator -> Bootstrap (via `shared`) |

No canonical "I'm done" signal name exists. New nodes must guess which convention to follow.

### 3.5 Casing Inconsistencies

- **File names**: PascalCase (correct) except `metadata.lua`, `definitions.lua`, `init.lua` (lowercase config files — acceptable convention).
- **Variable names**: Generally camelCase. No systematic violations found.
- **Module table keys**: Mix of camelCase (`computeHeightField`) and PascalCase (`BiomeInventory`) for different purposes. Acceptable.

## SECTION 5: SIMPLIFICATION OPPORTUNITIES

### S-5.1: Eliminate `_G.Warren` Global Access (HIGH)

**Current**: 20+ pipeline files access `_G.Warren.Dom`, `_G.Warren.ClassResolver`, etc. Three different bootstraps set this global differently.

**Proposed**: Inject framework services onto the pipeline `payload` (Houdini blackboard pattern, which is already the stated architecture). The orchestrator already has access to all framework services — pass them as `payload._system` or similar. Pipeline nodes read from payload instead of global.

**Benefit**: Eliminates 55+ global access sites, the bootstrap divergence bug, and the Lune mock requirement.

### S-5.2: Extract Material Validation to Shared Utility (LOW)

**Current**: `PortalBlender.lua` and `PortalRoomBuilder.lua` both contain identical 12-line material probe blocks at module load time.

**Proposed**: Extract to a shared `MaterialValidator.lua` utility that lazy-initializes on first call (not at require time), fixing both the duplication and the dual-runtime violation.

### S-5.3: Unify Completion Signal Convention (MEDIUM)

**Current**: `nodeComplete`, `buildPass`, `buildComplete`, `chunkDone` — four different "I'm done" signals.

**Proposed**: Standardize on one name (e.g., `nodeComplete`) for all pipeline node completion signals. The orchestrator's `_syncCall` already intercepts via `Out.Fire` monkey-patching, so the actual name could be anything — but consistency aids readability.

### S-5.4: Consolidate to One Bootstrap (HIGH)

**Current**: Two divergent server bootstraps — `Games/igw/Bootstrap.server.lua` and `Framework/v3/igw/Bootstrap.server.lua`.

**Proposed**: Delete `Framework/v3/igw/Bootstrap.server.lua` (legacy) and fix `Games/igw/Bootstrap.server.lua` to set `_G.Warren` unconditionally (if S-5.1 is not done first). Or better: implement S-5.1 and remove `_G.Warren` entirely.

### S-5.5: Replace `shared.dungeonReady` With Signal (MEDIUM)

**Current**: `shared.dungeonReady` is written with two incompatible schemas and consumed via busy-loop polling.

**Proposed**: Use a `BindableEvent` or the existing IPC system. Unify the schema to always include spawn data.

### S-5.6: Rename `WorldClient` -> `WorldBridge` (LOW)

Server-side node named "Client" is actively misleading. Rename to `WorldBridge` or `WorldServer`.

### S-5.7: Remove Transport/State Requires from init.lua (HIGH)

**Current**: `warren/src/init.lua` unconditionally requires Transport (6 files) and State (5 files) despite ADR-005.

**Proposed**: Delete the require lines. Delete the directories. This is the simplest change with the highest dead-code-removal payoff (~2,700 lines).
